# F5 — Pushing Privacy and Permanence Further

Concrete, repo-specific changes that would make Permawrite **more private** and
**more permanent** than the current design. Each item names what to
add/change/remove and where it lands in the codebase. This is a menu, not a
commitment — items graduate into [`ROADMAP.md`](./ROADMAP.md) when scheduled.

Baseline being improved on: consensus-enforced uniform ring-16 CLSAG,
Bulletproofs, stealth addresses, gamma decoy sampling, SPoRA storage proofs,
Pedersen-committed endowments, permanent tail emission
([`PRIVACY.md`](./PRIVACY.md), [`STORAGE.md`](./STORAGE.md),
[`ECONOMICS.md`](./ECONOMICS.md)). Known weaknesses this list responds to are
catalogued in [`PROBLEMS.md`](./PROBLEMS.md).

---

## Part 1 — More private

### P1. Ship Tier 2: raise the uniform ring size (change)

Raise `uniform_ring_size` in [`ConsensusParams`](../mfn-consensus/src/block/state.rs)
from 16 to 32–64 and migrate to **Bulletproof+** to claw back the transcript
bytes. Every ring member multiplies the anonymity set of every spend; ring 16
was chosen for Monero parity, not because it is optimal here. Requires a
coordinated wallet/CLI/WASM default bump (lane 5 owns that surface) and a
consensus version gate.

### P2. Ship Tier 3: One-out-of-Many over the whole UTXO set (add)

The Groth–Kohlweiss primitive already exists and is tested in
[`mfn_crypto::oom`](../mfn-crypto/src/oom.rs) but is not wired into
transactions. Wiring it in makes the anonymity set the **entire UTXO
history** instead of 16 decoys, which retires the gamma-calibration weakness
(PROBLEMS #9) outright: there is no decoy selection left to fingerprint. The
work is a UTXO-accumulator membership formulation plus the linkable
(Triptych-style) extension — wiring, not new cryptography.

### P3. Add Dandelion++ transaction relay (add)

Today a transaction's origin IP is visible to the first peers that see it;
`PRIVACY.md` explicitly punts network-layer privacy to "the wallet layer."
Implement Dandelion++ stem/fluff routing in `mfn-net` /
[`mfn-node/src/p2p_gossip.rs`](../mfn-node/src/p2p_gossip.rs) so the node that
broadcasts a tx is not identifiable as its author. This is the single largest
uncovered deanonymization surface in the current design.

### P4. Add Tor/arti transport support to mfnd (add)

Complement P3 with an optional onion-routed transport (the `arti` crate) for
P2P dials and RPC submission, configured in `mfnd` startup. Decoy rings hide
which UTXO you spent; they do nothing about a global network observer
correlating broadcast timing with home IP addresses.

### P5. Encrypt the payment graph edges: fixed-shape transactions (change)

Transaction topology (N inputs, M outputs) is public and is a known
fingerprinting vector. Enforce or pad to a small set of canonical shapes
(e.g. 2-in/2-out like Monero) in
[`verify_transaction`](../mfn-consensus/src/transaction/verify.rs) and make
wallets split/merge to fit. Uniformity is the same argument that justified
consensus ring-16: any degree of freedom the sender controls is a degree of
freedom that distinguishes senders.

> **Partially shipped (wallet layer).** The output-count half of this item is
> done: reference wallets never emit a single-output transaction — they pad to
> a two-output floor (`WALLET_MIN_TX_OUTPUTS = 2`). Remaining P5 work is (a)
> lifting that floor into `verify_transaction` so it is network-wide, and (b)
> canonicalizing input count and output ordering. See
> [`PRIVACY_HARDENING.md` §A1 and §B1/§B3](./PRIVACY_HARDENING.md).

### P6. Hide fees inside the balance equation (change)

Fees are currently public plaintext because the balance check needs
`fee · H`. Move to committed fees with a range proof and a per-block
aggregated fee disclosure (only the block-total fee is revealed, for the
coinbase check in `apply_block`). Removes the last plaintext amount from every
transaction. Nontrivial consensus change — touches
[`pedersen_balance`](../mfn-crypto/src/pedersen.rs) and the coinbase
settlement path.

### P7. Churn-resistant subaddress scanning / view tags (add)

Add Monero-style **view tags** (1-byte hint derived from the shared secret) to
outputs so light wallets can skip ~99% of scan work without changing the
privacy model. This is a privacy *enabler*: cheap scanning keeps users on
self-custodial light clients instead of handing view keys to third-party
scanning services. Lands in [`mfn_crypto::stealth`](../mfn-crypto/src/stealth.rs)
and the WASM light-client scan path.

### P8. Remove the LSAG legacy path from production builds (remove)

[`lsag.rs`](../mfn-crypto/src/lsag.rs) predates CLSAG and stays only as a
reference. Feature-gate it out of release binaries so no code path, test
harness, or future RPC can ever accept the weaker/larger variant on a
production chain. Shrinking the accepted-signature surface is a privacy
guarantee in itself.

### P9. Wallet-side anti-fingerprinting defaults (change)

Standardize everything a wallet chooses: output ordering, change position,
unlock-time (remove the field if unused), extra-field contents, and decoy
sampling seed handling. Any wallet-controlled byte that differs between
implementations partitions the anonymity set into "users of wallet X." Add a
canonical-encoding conformance test in `mfn-wallet` that all reference
frontends (CLI, WASM) must pass.

### P10. Formalize the authorship-key firewall (change)

`AUTHORSHIP.md` warns not to derive claiming keys from the stealth seed path;
make it structural instead of advisory. Give `ClaimingIdentity` a distinct,
incompatible derivation domain in `mfn-crypto` so a wallet *cannot* reuse the
financial seed, and add a lint/test that rejects cross-domain key material.

## Part 2 — More permanent

### PM1. Storage-operator bonds with slashing (add)

PROBLEMS #1: operators currently have no skin in the game — worst case they
just stop earning. Add a bonded-operator registry (mirroring the M1 validator
bond machinery in [`ECONOMICS.md` §9](./ECONOMICS.md#9-validator-bond-economics-m1-closed-loop)):
bond MFN to register, slash-to-treasury on provable non-storage (missed SPoRA
windows for data you claimed). Slashed bonds flow to the storage treasury, so
failed permanence directly funds surviving permanence. Lane 6 already owns
operator-bonding research.

### PM2. Raise and enforce minimum replication (change)

`replication` is carried in every
[`StorageCommitment`](./STORAGE.md#2-storagecommitment) but permanence today
is only as strong as the honest-operator count. Enforce a protocol minimum
(e.g. 3) at `apply_block`, and pay SPoRA rewards per *distinct operator* per
window rather than first-past-the-post, so replicas are economically real
instead of nominal.

### PM3. Replace the first-to-publish SPoRA race with windowed lotteries (change)

PROBLEMS #6: a pure latency race centralizes storage into whoever colocates
with producers, and centralized storage is fragile storage. Change proof
acceptance to a per-window VRF lottery among all valid proofs submitted in the
window, so geographically distributed operators earn proportional revenue.
Touches the proof pool ([`mfn-runtime/src/proof_pool.rs`](../mfn-runtime/src/proof_pool.rs))
and Phase 4 of `apply_block`.

### PM4. Erasure coding below the replication layer (add)

Whole-copy replication is the bluntest redundancy instrument. Add optional
Reed–Solomon striping at the chunking layer
([`STORAGE.md` §1](./STORAGE.md#1-chunking--content-addressing)) so a file
survives the loss of any k-of-n operators at a fraction of full-copy cost.
Endowment pricing in [`STORAGE_COST_MODEL.md`](./STORAGE_COST_MODEL.md) then
buys strictly more durability per MFN.

### PM5. Diversify treasury inflow away from the r = 0 fee dependency (change)

PROBLEMS #2: with the r = 0 default, permanence leans heavily on continuous
privacy-fee volume. Two concrete levers: (a) route a small basis-point share
of the **tail emission** directly to the storage treasury (a one-line change
in the coinbase split, but a monetary-policy decision); (b) let endowments
opt into over-funding tiers whose surplus seeds the treasury buffer. Both
shrink the fee-drought scenario space in
[`ECONOMICS.md` §5](./ECONOMICS.md#5-treasury-dynamics--worked-scenarios).

### PM6. Cover `utxo_root` (and storage roots) with the finality signature (change)

PROBLEMS #12: finality currently attests ordering, not state. Extend the
committee-signed digest to include `utxo_root` and the storage-proof root so a
finalized block *cryptographically commits* to the storage state. Permanence
you cannot verify from a light client is permanence you are taking on faith —
this makes the guarantee checkable from
[`mfn-light`](../mfn-light/src/chain.rs) and the WASM client.

### PM7. Historical-chunk availability sampling for light clients (add)

Add a light-client protocol that randomly samples old chunks (request +
Merkle-verify against the committed `data_root`) so *anyone* can continuously
audit that the network still holds the data — not just consensus nodes paying
SPoRA. Publishing failed-sample evidence on-chain becomes the trigger for PM1
slashing. Natural extension of the
[`STORAGE_ACCESSIBILITY.md`](./STORAGE_ACCESSIBILITY.md) prove+serve work.

### PM8. State-growth relief: checkpointed chain pruning with permanence carve-out (change)

PROBLEMS #7: linear state growth is itself a permanence risk — if running a
full node becomes exotic, the operator set shrinks. Extend the
[`chain_checkpoint`](../mfn-consensus/src/chain_checkpoint) machinery so
nodes can prune spent transaction data behind a finalized checkpoint while
**never** pruning storage commitments, endowment state, or committed data.
The chain's memory of *money* can be summarized; its memory of *data* must
not be.

### PM9. Multi-decade key-rotation and PQ migration plan (add)

Permanence horizons (see [`SUPPLY_CURVE.md`](./SUPPLY_CURVE.md) deep-time
section) exceed any sane confidence interval for ed25519. Write and commit a
consensus-versioned migration path: hybrid signatures (ed25519 + ML-DSA) for
validator/finality keys first, then a PQ successor for CLSAG (research-grade
today). The point of doing it now is wire-format headroom: version bytes and
length fields in [`mfn-consensus`](../mfn-consensus/src/lib.rs) codecs so the
migration is a soft fork, not an archaeology project.

### PM10. Genesis-anchored external archive mirrors (add)

Defense in depth for the doomsday case: a `scripts/`-level tool that exports
the canonical chain + chunk archive in a self-verifying format (headers +
finality signatures + data roots) suitable for deposit in offline/third-party
archives. Anyone holding the export and the genesis hash can verify every
byte with no live network. Cheap to build on the existing archive dry-run
tooling; turns "the network survives" from a single point of truth into one
of several.

## Part 3 — Frontier: privacy beyond any deployed system

These are the items that separate "Monero-class privacy with permanence" from
**strictly stronger than anything running today**.

### P11. Full-chain membership proofs (curve trees) — retire rings entirely (add)

The endgame of the P2 direction. Instead of OoM over a flat UTXO list,
maintain a **curve-tree accumulator** over all outputs ever created and prove
membership with a zero-knowledge proof (the Monero FCMP++ research line).
Every spend's anonymity set becomes *every output in chain history*, with
`O(log N)` proof size and no per-tx decoy selection at all. Kills the entire
class of statistical ring attacks — timing, chain-reaction, poisoned-output
tracing — cryptographically rather than statistically. The accumulator slots
naturally into the UTXO bookkeeping `apply_block` already maintains.

### P12. Forward privacy against harvest-now-decrypt-later (add)

Pedersen commitments are **information-theoretically hiding** — amounts stay
secret even against a future quantum adversary. But stealth-address detection
rests on ECDH: an adversary archiving the chain today and breaking discrete
log in 2060 can retroactively link every historical output to its recipient.
For a permanence chain this is not hypothetical — the data *will* still be
there. Fix: hybrid the shared-secret derivation in
[`mfn_crypto::stealth`](../mfn-crypto/src/stealth.rs) with an ML-KEM
encapsulation (`s_shared = H(ECDH || ML-KEM secret)`), so unlinkability holds
if *either* assumption survives. No other privacy chain ships this today;
Permawrite's forever horizon makes it uniquely obligatory here.

### P13. Private reads: oblivious chunk retrieval (add)

Uploads are ring-signed and amount-private, but **reads are naked**: a chunk
server learns exactly which `data_root` you fetched, which is often more
sensitive than having stored it (think: who is reading the leaked archive).
Add a private-information-retrieval tier to the prove+serve path
([`STORAGE_ACCESSIBILITY.md`](./STORAGE_ACCESSIBILITY.md)): batched
single-server PIR over hot chunk sets, plus a cheaper baseline of oblivious
mixing across replicas (fetch k decoy chunks per real chunk from distinct
operators). Censorship-resistant storage is incomplete if reading it puts you
on a list.

### P14. Oblivious light-client sync (add)

View tags (P7) make scanning cheap; this makes *delegated* scanning safe.
Oblivious message retrieval lets an untrusted server find your outputs
without learning which outputs are yours. Light clients on phones are the
realistic majority forever — if their sync path leaks the recipient set, the
on-chain privacy stack protects a minority. Lands as an opt-in protocol next
to the WASM light client (`mfn-wasm`, [`mfn-light`](../mfn-light/src/chain.rs)).

### P15. Size-bucketed storage commitments (change)

`size_bytes` is public in every
[`StorageCommitment`](./STORAGE.md#2-storagecommitment), and exact file size
is a strong fingerprint (a 1,442,839-byte upload is trivially matchable to a
known document). Pad uploads to a small set of size buckets (powers of two,
or 1.5× steps) at the chunking layer and price endowments on the bucket.
Costs a few percent of storage; removes the highest-entropy public metadata
field the chain still exposes.

### P16. Cover traffic and uniform broadcast timing (add)

Even with Dandelion++ (P3) and Tor (P4), a global passive observer can
correlate *when* wallets come online with *when* transactions appear. Give
`mfnd` and reference wallets a constant-rate cover regime: scheduled
broadcast slots, dummy fetches, and padded message sizes on the P2P wire
(`mfn-net`). Metadata resistance at the network layer is what pushes the
threat model from "state-level chain analyst" to "global passive adversary" —
the strongest standard there is.

## Part 4 — Frontier: permanence beyond any deployed system

### PM11. Data-availability sampling at consensus (add)

Erasure-code every block's storage payload and have light clients verify
availability by random sampling (the Celestia construction). Today only full
nodes and SPoRA-paying operators can attest data exists; with DAS, **every
phone running the WASM client** becomes an availability auditor, and
withholding data becomes detectable by the weakest participant. Combined with
PM6 (finality over state roots), a light client gets end-to-end proof that
finalized data is really out there — no other permanence chain gives its
lightest clients that guarantee.

### PM12. Self-healing replication market (add)

Permanence should not depend on anyone noticing decay. When the count of
distinct operators proving a `data_root` in a window drops below its
replication target, the protocol automatically opens a **repair bounty**
funded by that file's endowment: elevated reward multiplier for the first new
operators to prove the under-replicated data. Turns PM2's static minimum into
a closed control loop — the chain detects erosion via the proof pool it
already runs ([`mfn-runtime/src/proof_pool.rs`](../mfn-runtime/src/proof_pool.rs))
and bids its own repair.

### PM13. Constitutional invariants — ossify what must never change (add)

Permanence is a promise about *governance*, not just disks. Define a small
set of constitutional invariants that reference clients hard-refuse to
violate regardless of any future fork or parameter vote: `tail_emission > 0`
(already validated in [`emission.rs`](../mfn-consensus/src/emission.rs)),
`uniform_ring_size >= 16`, storage state is never prunable, endowment funds
are never confiscatable. Enforce them in `validate_emission_params`-style
genesis/upgrade checks and document them as the fork-legitimacy test: a chain
that violates one is, by definition, not Permawrite. Bitcoin's 21M cap shows
ossification is a feature; here it is codified rather than cultural.

### PM14. Second independent client implementation (add)

The largest single systemic risk to a forever chain is client monoculture: one
memory-safety bug or consensus edge case in `mfn-consensus` is a
chain-halting — or chain-splitting — event. Fund and maintain an independent
implementation (different language, different team) validated continuously
against the Rust reference via differential fuzzing of `apply_block` and the
codec layer (the [`docs/interop`](./interop) vectors are the seed corpus).
Ethereum treats client diversity as existential; a permanence chain should
hold itself to at least that bar.

### PM15. Formally verified consensus core (add)

Push the invariants the M5 proptests sample into machine-checked proofs:
no-inflation (coinbase == subsidy + fees + storage payouts, exactly),
endowment conservation, and key-image uniqueness under `apply_block`. Rust
has practical tooling (Kani model checking, Creusot/Verus deductive proofs)
that can target the pure functions in
[`mfn-consensus/src/block`](../mfn-consensus/src/block/state.rs) without
rewriting them. A proof is the only artifact stronger than a test suite on a
chain that intends to outlive its authors.

### PM16. Broadcast-path independence: checkpoints over satellite/radio (add)

If the internet partitions — nationally or globally — the chain's canonical
tip should still be receivable one-way. Publish signed chain checkpoints
(the [`chain_checkpoint`](../mfn-consensus/src/chain_checkpoint) format is
already self-verifying) over satellite broadcast, shortwave, and other
non-IP channels, and teach `mfn-light` to accept them as subjective
checkpoints. Cheap: the artifact already exists; this is a distribution
program, not a protocol change. Pairs with PM10's offline archives to make
the chain reconstructible from a cold start on any continent.

### PM17. Endow the protocol itself (add)

Data has endowments; the software that keeps it alive does not. Route a small
treasury stream into a protocol-maintenance endowment that pays for exactly
the things markets under-fund on century timescales: security audits, PM14's
second client, PM9's PQ migration, archive refreshes. Spend policy can be as
conservative as "N-of-M maintainer multisig, capped annual drawdown" at
first. A chain that promises forever must fund forever — permanence of the
*maintenance loop* is as load-bearing as permanence of the bytes.

## Prioritization sketch

| Impact / effort | Privacy | Permanence |
|---|---|---|
| High impact, moderate effort | P3 (Dandelion++), P7 (view tags), P15 (size buckets) | PM1 (operator bonds), PM2 (real replication), PM13 (constitutional invariants) |
| High impact, high effort | P2 (Tier 3 OoM), P6 (hidden fees), P11 (full-chain membership), P12 (PQ stealth hybrid) | PM3 (proof lottery), PM6 (finality covers state), PM11 (DAS), PM12 (self-healing market) |
| Cheap wins | P8 (drop LSAG), P10 (key firewall) | PM10 (archive export), PM5a (tail-to-treasury split), PM16 (broadcast checkpoints) |
| Long horizon | P5 (uniform tx shapes), P13 (private reads), P14 (oblivious sync), P16 (cover traffic) | PM9 (PQ migration), PM8 (pruning carve-out), PM14 (second client), PM15 (formal proofs), PM17 (protocol endowment) |

The "more robust than anything on earth" bar, concretely: P11 + P12 give
anonymity sets of the entire chain history that survive a future quantum
archive attack; P13 + P16 extend privacy from writing to *reading* under a
global passive adversary; PM11 + PM6 let the weakest client verify the data
exists; PM13 + PM14 + PM17 make the guarantee survive governance capture,
client monoculture, and maintainer attrition. No deployed system today —
Monero, Zcash, Arweave, Filecoin, or any L1 — clears all four rows at once.

Doctrine check: every item above strengthens ring policy, endowment
enforcement, or SPoRA verification — none weakens them. Items that change
consensus (P1, P5, P6, P11, P12, P15, PM2, PM3, PM5, PM6, PM11, PM12, PM13)
require a version gate and the full M5-style test treatment before they touch
`main`.
