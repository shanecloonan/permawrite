# F5 — Pushing Privacy and Permanence Further

Concrete, repo-specific changes that would make Permawrite **more private** and
**more permanent** than the current design. Each item names what to
add/change/remove and where it lands in the codebase. This is a menu, not a
commitment — items graduate into [`ROADMAP.md`](./ROADMAP.md) when scheduled.
The menu is organized in fourteen parts (Parts 1–4 baseline + frontier, Part 5 fused,
Parts 6–7 depth, Parts 8–10 adversarial / ops / research, Parts 11–12 network +
measurement, Parts 13–14 sovereignty + scale) — **120 items** total; see the
index below.

Baseline being improved on: consensus-enforced uniform ring-16 CLSAG,
Bulletproofs, stealth addresses, gamma decoy sampling, SPoRA storage proofs,
Pedersen-committed endowments, permanent tail emission
([`PRIVACY.md`](./PRIVACY.md), [`STORAGE.md`](./STORAGE.md),
[`ECONOMICS.md`](./ECONOMICS.md)). Known weaknesses this list responds to are
catalogued in [`PROBLEMS.md`](./PROBLEMS.md).

## Part 1 — More private

### P1. Ship Tier 2: raise the uniform ring size (change)

Raise `uniform_ring_size` in [`ConsensusParams`](../mfn-consensus/src/block/state.rs)
from 16 to 32–64 and migrate to **Bulletproof+** to claw back the transcript
bytes. Every ring member multiplies the anonymity set of every spend; ring 16
was chosen for Monero parity, not because it is optimal here. Requires a
coordinated wallet/CLI/WASM default bump (lane 5 owns that surface) and a
consensus version gate.

### P2. Ship Tier 3: One-out-of-Many over the whole UTXO set (add)

The Groth–Kohlweiss primitive already exists and is tested in
[`mfn_crypto::oom`](../mfn-crypto/src/oom.rs) but is not wired into
transactions. Wiring it in makes the anonymity set the **entire UTXO
history** instead of 16 decoys, which retires the gamma-calibration weakness
(PROBLEMS #9) outright: there is no decoy selection left to fingerprint. The
work is a UTXO-accumulator membership formulation plus the linkable
(Triptych-style) extension — wiring, not new cryptography.

### P3. Add Dandelion++ transaction relay (add)

Today a transaction's origin IP is visible to the first peers that see it;
`PRIVACY.md` explicitly punts network-layer privacy to "the wallet layer."
Implement Dandelion++ stem/fluff routing in `mfn-net` /
[`mfn-node/src/p2p_gossip.rs`](../mfn-node/src/p2p_gossip.rs) so the node that
broadcasts a tx is not identifiable as its author. This is the single largest
uncovered deanonymization surface in the current design.

### P4. Add Tor/arti transport support to mfnd (add)

Complement P3 with an optional onion-routed transport (the `arti` crate) for
P2P dials and RPC submission, configured in `mfnd` startup. Decoy rings hide
which UTXO you spent; they do nothing about a global network observer
correlating broadcast timing with home IP addresses.

### P5. Encrypt the payment graph edges: fixed-shape transactions (change)

Transaction topology (N inputs, M outputs) is public and is a known
fingerprinting vector. Enforce or pad to a small set of canonical shapes
(e.g. 2-in/2-out like Monero) in
[`verify_transaction`](../mfn-consensus/src/transaction/verify.rs) and make
wallets split/merge to fit. Uniformity is the same argument that justified
consensus ring-16: any degree of freedom the sender controls is a degree of
freedom that distinguishes senders.

> **Mostly shipped.** The output-count half is now **consensus law**, not
> just wallet courtesy: `RingPolicy.min_output_count`
> (`MIN_TX_OUTPUTS_UNIFORM_TIER = 2`) is enforced in `verify_transaction`
> whenever the uniform-ring tier is active, guarding both mempool admission
> and `apply_block`. The floor is derived from `uniform_ring_size != 0`, so
> checkpoint serialization is unchanged. Output *ordering* is canonicalized
> (B3 shuffle + conformance suite). Remaining P5 work: canonical input-count
> shapes (split/merge to fixed N-in), which needs a fee/UX design pass. See
> [`PRIVACY_HARDENING.md` §A1/§B1/§B3](./PRIVACY_HARDENING.md).

### P6. Hide fees inside the balance equation (change)

Fees are currently public plaintext because the balance check needs
`fee · H`. Move to committed fees with a range proof and a per-block
aggregated fee disclosure (only the block-total fee is revealed, for the
coinbase check in `apply_block`). Removes the last plaintext amount from every
transaction. Nontrivial consensus change — touches
[`pedersen_balance`](../mfn-crypto/src/pedersen.rs) and the coinbase
settlement path.

### P7. Churn-resistant subaddress scanning / view tags (add) — **shipped (B9)**

Monero-style **view tags** (1-byte hint derived from the shared secret) are on
the tx v2 wire; light wallets skip ~256× of scan work before stealth-detect.
Derivation lives in [`mfn_crypto::stealth`](../mfn-crypto/src/stealth.rs);
encode on send and filter on scan in the reference wallet
([`mfn-wallet/src/scan.rs`](../mfn-wallet/src/scan.rs)). Legacy v1 txs remain
valid at ingress.

### P8. Remove the LSAG legacy path from production builds (remove)

[`lsag.rs`](../mfn-crypto/src/lsag.rs) predates CLSAG and stays only as a
reference. Feature-gate it out of release binaries so no code path, test
harness, or future RPC can ever accept the weaker/larger variant on a
production chain. Shrinking the accepted-signature surface is a privacy
guarantee in itself.

> **Shipped.** `mfn_crypto::lsag` (and the not-yet-wired `oom` module, per
> the same argument) compile only under `cfg(test)` or the non-default
> `lsag` / `oom` cargo features. No dependent crate enables them, so
> release binaries carry CLSAG as the sole accepted ring-signature
> surface. See [`PRIVACY_HARDENING.md` §B5](./PRIVACY_HARDENING.md).

### P9. Wallet-side anti-fingerprinting defaults (change)

Standardize everything a wallet chooses: output ordering, change position,
unlock-time (remove the field if unused), extra-field contents, and decoy
sampling seed handling. Any wallet-controlled byte that differs between
implementations partitions the anonymity set into "users of wallet X." Add a
canonical-encoding conformance test in `mfn-wallet` that all reference
frontends (CLI, WASM) must pass.

> **Shipped.** Output ordering: `spend::build_transfer` Fisher–Yates-shuffles
> output specs with the plan RNG, so change position carries no signal.
> Unlock-time: the wire format has no such field (strongest form of
> "remove the field if unused"). Conformance:
> [`mfn-wallet/tests/canonical_conformance.rs`](../mfn-wallet/tests/canonical_conformance.rs)
> pins version, empty-`extra` default, uniform ring-16 (pinned equal to the
> consensus production policy), the two-output floor, real `enc_amount`
> ciphertexts, and byte-canonical encoding for both transfers and uploads.
> All frontends funnel through the two pinned constructors, so the suite
> covers CLI + WASM by construction. Production RNG: [`production_tx_rng`]
> is the normative OS CSPRNG alias; reference frontends wire it and the
> conformance suite source-scans against `seeded_rng` in production paths.
> See [`PRIVACY_HARDENING.md` §B3](./PRIVACY_HARDENING.md).

### P10. Formalize the authorship-key firewall (change)

`AUTHORSHIP.md` warns not to derive claiming keys from the stealth seed path;
make it structural instead of advisory. Give `ClaimingIdentity` a distinct,
incompatible derivation domain in `mfn-crypto` so a wallet *cannot* reuse the
financial seed, and add a lint/test that rejects cross-domain key material.

> **Shipped.** `mfn_crypto::authorship::derive_claiming_keypair` is the
> canonical domain-separated derivation; `ClaimingIdentity`'s only
> constructor delegates to it, and the wallet claim paths reject any
> claiming pubkey that collides with the wallet's view/spend pubkeys
> (`WalletError::ClaimKeyReusesWalletKey`). See
> [`PRIVACY_HARDENING.md` §B10](./PRIVACY_HARDENING.md).

## Part 2 — More permanent

### PM1. Storage-operator bonds with slashing (add)

PROBLEMS #1: operators currently have no skin in the game — worst case they
just stop earning. Add a bonded-operator registry (mirroring the M1 validator
bond machinery in [`ECONOMICS.md` §9](./ECONOMICS.md#9-validator-bond-economics-m1-closed-loop)):
bond MFN to register, slash-to-treasury on provable non-storage (missed SPoRA
windows for data you claimed). Slashed bonds flow to the storage treasury, so
failed permanence directly funds surviving permanence. Lane 6 already owns
operator-bonding research.

### PM2. Raise and enforce minimum replication (change)

`replication` is carried in every
[`StorageCommitment`](./STORAGE.md#2-storagecommitment) but permanence today
is only as strong as the honest-operator count. Enforce a protocol minimum
(e.g. 3) at `apply_block`, and pay SPoRA rewards per *distinct operator* per
window rather than first-past-the-post, so replicas are economically real
instead of nominal.

### PM3. Replace the first-to-publish SPoRA race with windowed lotteries (change)

PROBLEMS #6: a pure latency race centralizes storage into whoever colocates
with producers, and centralized storage is fragile storage. Change proof
acceptance to a per-window VRF lottery among all valid proofs submitted in the
window, so geographically distributed operators earn proportional revenue.
Touches the proof pool ([`mfn-runtime/src/proof_pool.rs`](../mfn-runtime/src/proof_pool.rs))
and Phase 4 of `apply_block`.

### PM4. Erasure coding below the replication layer (add)

Whole-copy replication is the bluntest redundancy instrument. Add optional
Reed–Solomon striping at the chunking layer
([`STORAGE.md` §1](./STORAGE.md#1-chunking--content-addressing)) so a file
survives the loss of any k-of-n operators at a fraction of full-copy cost.
Endowment pricing in [`STORAGE_COST_MODEL.md`](./STORAGE_COST_MODEL.md) then
buys strictly more durability per MFN.

### PM5. Diversify treasury inflow away from the r = 0 fee dependency (change)

PROBLEMS #2: with the r = 0 default, permanence leans heavily on continuous
privacy-fee volume. Two concrete levers: (a) route a small basis-point share
of the **tail emission** directly to the storage treasury (a one-line change
in the coinbase split, but a monetary-policy decision); (b) let endowments
opt into over-funding tiers whose surplus seeds the treasury buffer. Both
shrink the fee-drought scenario space in
[`ECONOMICS.md` §5](./ECONOMICS.md#5-treasury-dynamics--worked-scenarios).

### PM6. Cover `utxo_root` (and storage roots) with the finality signature (change)

PROBLEMS #12: finality currently attests ordering, not state. Extend the
committee-signed digest to include `utxo_root` and the storage-proof root so a
finalized block *cryptographically commits* to the storage state. Permanence
you cannot verify from a light client is permanence you are taking on faith —
this makes the guarantee checkable from
[`mfn-light`](../mfn-light/src/chain.rs) and the WASM client.

### PM7. Historical-chunk availability sampling for light clients (add)

Add a light-client protocol that randomly samples old chunks (request +
Merkle-verify against the committed `data_root`) so *anyone* can continuously
audit that the network still holds the data — not just consensus nodes paying
SPoRA. Publishing failed-sample evidence on-chain becomes the trigger for PM1
slashing. Natural extension of the
[`STORAGE_ACCESSIBILITY.md`](./STORAGE_ACCESSIBILITY.md) prove+serve work.

### PM8. State-growth relief: checkpointed chain pruning with permanence carve-out (change)

PROBLEMS #7: linear state growth is itself a permanence risk — if running a
full node becomes exotic, the operator set shrinks. Extend the
[`chain_checkpoint`](../mfn-consensus/src/chain_checkpoint) machinery so
nodes can prune spent transaction data behind a finalized checkpoint while
**never** pruning storage commitments, endowment state, or committed data.
The chain's memory of *money* can be summarized; its memory of *data* must
not be.

### PM9. Multi-decade key-rotation and PQ migration plan (add)

Permanence horizons (see [`SUPPLY_CURVE.md`](./SUPPLY_CURVE.md) deep-time
section) exceed any sane confidence interval for ed25519. Write and commit a
consensus-versioned migration path: hybrid signatures (ed25519 + ML-DSA) for
validator/finality keys first, then a PQ successor for CLSAG (research-grade
today). The point of doing it now is wire-format headroom: version bytes and
length fields in [`mfn-consensus`](../mfn-consensus/src/lib.rs) codecs so the
migration is a soft fork, not an archaeology project.

> **Shipped (plan committed).** [`PQ_MIGRATION.md`](./PQ_MIGRATION.md) is
> the normative migration path: surface inventory with risk classes
> (retroactive privacy vs. forge-later integrity), three consensus-versioned
> phases (A: hybrid ML-KEM stealth = P12; B: hybrid BLS + ML-DSA
> validator/finality keys; C: research-gated CLSAG successor), a
> wire-format headroom audit showing each step is a soft fork today, and
> the codec discipline reviewers must hold (every new crypto object enters
> the wire as a length-prefixed blob behind a version field). Code work
> lands per-phase later; the plan and headroom audit are the PM9
> deliverable.

### PM10. Genesis-anchored external archive mirrors (add)

Defense in depth for the doomsday case: a `scripts/`-level tool that exports
the canonical chain + chunk archive in a self-verifying format (headers +
finality signatures + data roots) suitable for deposit in offline/third-party
archives. Anyone holding the export and the genesis hash can verify every
byte with no live network. Cheap to build on the existing archive dry-run
tooling; turns "the network survives" from a single point of truth into one
of several.

> **Shipped.** `mfnd archive-export --archive-dir DIR` writes the canonical
> block log plus every locally-complete, Merkle-verified chunk set (and a
> `manifest.json` pinning genesis/tip ids) into a portable directory;
> `mfnd archive-verify` replays the archive from the genesis spec through
> the **full consensus state-transition function** and re-derives each
> chunk set's Merkle root against the anchored `data_root` — no live
> network, no trusted checksums. The archive reuses the `mfn-store`
> filesystem formats, so it doubles as a cold-start bootstrap for a fresh
> `--store fs` node. See
> [`mfn-node/src/archive_export.rs`](../mfn-node/src/archive_export.rs).

## Part 3 — Frontier: privacy beyond any deployed system

These are the items that separate "Monero-class privacy with permanence" from
**strictly stronger than anything running today**.

### P11. Full-chain membership proofs (curve trees) — retire rings entirely (add)

The endgame of the P2 direction. Instead of OoM over a flat UTXO list,
maintain a **curve-tree accumulator** over all outputs ever created and prove
membership with a zero-knowledge proof (the Monero FCMP++ research line).
Every spend's anonymity set becomes *every output in chain history*, with
`O(log N)` proof size and no per-tx decoy selection at all. Kills the entire
class of statistical ring attacks — timing, chain-reaction, poisoned-output
tracing — cryptographically rather than statistically. The accumulator slots
naturally into the UTXO bookkeeping `apply_block` already maintains.

### P12. Forward privacy against harvest-now-decrypt-later (add)

Pedersen commitments are **information-theoretically hiding** — amounts stay
secret even against a future quantum adversary. But stealth-address detection
rests on ECDH: an adversary archiving the chain today and breaking discrete
log in 2060 can retroactively link every historical output to its recipient.
For a permanence chain this is not hypothetical — the data *will* still be
there. Fix: hybrid the shared-secret derivation in
[`mfn_crypto::stealth`](../mfn-crypto/src/stealth.rs) with an ML-KEM
encapsulation (`s_shared = H(ECDH || ML-KEM secret)`), so unlinkability holds
if *either* assumption survives. No other privacy chain ships this today;
Permawrite's forever horizon makes it uniquely obligatory here.

### P13. Private reads: oblivious chunk retrieval (add)

Uploads are ring-signed and amount-private, but **reads are naked**: a chunk
server learns exactly which `data_root` you fetched, which is often more
sensitive than having stored it (think: who is reading the leaked archive).
Add a private-information-retrieval tier to the prove+serve path
([`STORAGE_ACCESSIBILITY.md`](./STORAGE_ACCESSIBILITY.md)): batched
single-server PIR over hot chunk sets, plus a cheaper baseline of oblivious
mixing across replicas (fetch k decoy chunks per real chunk from distinct
operators). Censorship-resistant storage is incomplete if reading it puts you
on a list.

### P14. Oblivious light-client sync (add)

View tags (P7) make scanning cheap; this makes *delegated* scanning safe.
Oblivious message retrieval lets an untrusted server find your outputs
without learning which outputs are yours. Light clients on phones are the
realistic majority forever — if their sync path leaks the recipient set, the
on-chain privacy stack protects a minority. Lands as an opt-in protocol next
to the WASM light client (`mfn-wasm`, [`mfn-light`](../mfn-light/src/chain.rs)).

### P15. Size-bucketed storage commitments (change)

`size_bytes` is public in every
[`StorageCommitment`](./STORAGE.md#2-storagecommitment), and exact file size
is a strong fingerprint (a 1,442,839-byte upload is trivially matchable to a
known document). Pad uploads to a small set of size buckets (powers of two,
or 1.5× steps) at the chunking layer and price endowments on the bucket.

> **Shipped (B13).** Reference uploads pad to the next power-of-two via
> [`storage_size_bucket`](../mfn-storage/src/commitment.rs) /
> [`pad_to_storage_size_bucket`](../mfn-storage/src/commitment.rs) in
> `build_storage_upload`; consensus rejects non-bucket `size_bytes` at
> apply/mempool; operators persist `UploadArtifacts.anchored_payload`. See
> [`PRIVACY_HARDENING.md` §B13](./PRIVACY_HARDENING.md).

### P16. Cover traffic and uniform broadcast timing (add)

Even with Dandelion++ (P3) and Tor (P4), a global passive observer can
correlate *when* wallets come online with *when* transactions appear. Give
`mfnd` and reference wallets a constant-rate cover regime: scheduled
broadcast slots, dummy fetches, and padded message sizes on the P2P wire
(`mfn-net`). Metadata resistance at the network layer is what pushes the
threat model from "state-level chain analyst" to "global passive adversary" —
the strongest standard there is.

## Part 4 — Frontier: permanence beyond any deployed system

### PM11. Data-availability sampling at consensus (add)

Erasure-code every block's storage payload and have light clients verify
availability by random sampling (the Celestia construction). Today only full
nodes and SPoRA-paying operators can attest data exists; with DAS, **every
phone running the WASM client** becomes an availability auditor, and
withholding data becomes detectable by the weakest participant. Combined with
PM6 (finality over state roots), a light client gets end-to-end proof that
finalized data is really out there — no other permanence chain gives its
lightest clients that guarantee.

### PM12. Self-healing replication market (add)

Permanence should not depend on anyone noticing decay. When the count of
distinct operators proving a `data_root` in a window drops below its
replication target, the protocol automatically opens a **repair bounty**
funded by that file's endowment: elevated reward multiplier for the first new
operators to prove the under-replicated data. Turns PM2's static minimum into
a closed control loop — the chain detects erosion via the proof pool it
already runs ([`mfn-runtime/src/proof_pool.rs`](../mfn-runtime/src/proof_pool.rs))
and bids its own repair.

### PM13. Constitutional invariants — ossify what must never change (add)

Permanence is a promise about *governance*, not just disks. Define a small
set of constitutional invariants that reference clients hard-refuse to
violate regardless of any future fork or parameter vote: `tail_emission > 0`
(already validated in [`emission.rs`](../mfn-consensus/src/emission.rs)),
`uniform_ring_size >= 16`, storage state is never prunable, endowment funds
are never confiscatable. Enforce them in `validate_emission_params`-style
genesis/upgrade checks and document them as the fork-legitimacy test: a chain
that violates one is, by definition, not Permawrite. Bitcoin's 21M cap shows
ossification is a feature; here it is codified rather than cultural.

> **Shipped (parameter invariants).**
> [`mfn_consensus::constitution`](../mfn-consensus/src/constitution.rs)
> defines `validate_constitution`: `tail_emission > 0`, uniform CLSAG rings
> `>= 16` (uniformity may not be disabled; raising is legal), and
> well-formed endowment pricing. Enforced on every operator-supplied
> genesis spec (`genesis_config_from_json_bytes`), i.e. every `mfnd`
> startup. The code-path invariants (storage state never prunable,
> endowments never confiscatable) are documented there as review-time
> constitution violations and pinned by the M5 proptest suite.

### PM14. Second independent client implementation (add)

The largest single systemic risk to a forever chain is client monoculture: one
memory-safety bug or consensus edge case in `mfn-consensus` is a
chain-halting — or chain-splitting — event. Fund and maintain an independent
implementation (different language, different team) validated continuously
against the Rust reference via differential fuzzing of `apply_block` and the
codec layer (the [`docs/interop`](./interop) vectors are the seed corpus).
Ethereum treats client diversity as existential; a permanence chain should
hold itself to at least that bar.

### PM15. Formally verified consensus core (add)

Push the invariants the M5 proptests sample into machine-checked proofs:
no-inflation (coinbase == subsidy + fees + storage payouts, exactly),
endowment conservation, and key-image uniqueness under `apply_block`. Rust
has practical tooling (Kani model checking, Creusot/Verus deductive proofs)
that can target the pure functions in
[`mfn-consensus/src/block`](../mfn-consensus/src/block/state.rs) without
rewriting them. A proof is the only artifact stronger than a test suite on a
chain that intends to outlive its authors.

### PM16. Broadcast-path independence: checkpoints over satellite/radio (add)

If the internet partitions — nationally or globally — the chain's canonical
tip should still be receivable one-way. Publish signed chain checkpoints
(the [`chain_checkpoint`](../mfn-consensus/src/chain_checkpoint) format is
already self-verifying) over satellite broadcast, shortwave, and other
non-IP channels, and teach `mfn-light` to accept them as subjective
checkpoints. Cheap: the artifact already exists; this is a distribution
program, not a protocol change. Pairs with PM10's offline archives to make
the chain reconstructible from a cold start on any continent.

### PM17. Endow the protocol itself (add)

Data has endowments; the software that keeps it alive does not. Route a small
treasury stream into a protocol-maintenance endowment that pays for exactly
the things markets under-fund on century timescales: security audits, PM14's
second client, PM9's PQ migration, archive refreshes. Spend policy can be as
conservative as "N-of-M maintainer multisig, capped annual drawdown" at
first. A chain that promises forever must fund forever — permanence of the
*maintenance loop* is as load-bearing as permanence of the bytes.

## Part 5 — Fused bar: privacy and permanence as one guarantee

These are the highest-leverage items **because money and data share one chain**.
No Monero fork can do the permanence half; no Arweave fork can do the privacy
half. This section is the "absolute bar" checklist.

### F1. Consensus-bind endowment Pedersen openings (`B-11`) (change)

Today the chain checks that the upload tx's **fee share** covers
`required_endowment`, but the `StorageCommitment.endowment` field is a bare
Pedersen commitment — an uploader can commit to an opening that does **not**
match the protocol price while still passing geometry and fee gates. That is
the highest remaining permanence gap after M5.49/M7.12
([`PERMANENCE_HARDENING.md`](./PERMANENCE_HARDENING.md)). Fix: at
`apply_block`, require either (a) a range proof that the committed endowment
lies in `[required, MAX]`, or (b) an opening reveal bound to the same tx that
paid the fee. Without this, "paid forever" is a social convention, not a
consensus fact. Touches
[`mfn-storage/src/endowment.rs`](../mfn-storage/src/endowment.rs) and the
storage-anchoring walk in `apply_block`.

### F2. Default encrypt-before-upload in every reference frontend (change)

`STORAGE.md` is explicit: plaintext uploads are public forever. Permanence
without confidentiality is a liability for whistleblowers, key material, and
medical records — the exact users the vision targets. Make client-side
encryption (random per-file key, AEAD, key wrapped to recipient stealth
address or out-of-band) the **default path** in `mfn-wallet`, `mfn-cli`, and
`mfn-wasm`, with a scary opt-out for public artifacts only. The chain still
stores deterministic SPoRA-verifiable bytes; users choose whether those bytes
are meaningful. Pairs with P13 (private reads): encryption at rest on disk is
useless if fetching the ciphertext deanonymizes the reader.

### F3. Validity-attested finality — vote only after `apply_block` (change)

PROBLEMS #11 / [`SECURITY_CONSIDERATIONS.md` §2](./SECURITY_CONSIDERATIONS.md#2-what-a-finalized-header-does--and-does-not--prove):
committee signatures attest header bytes, not state-transition validity. Light
clients therefore inherit a strictly weaker guarantee than full nodes. Require
each voter to include a hash of `(pre_state_root, post_state_root,
apply_block_result)` in the vote payload, or move to a two-phase vote:
propose → execute locally → attest execution digest. Makes "finalized" mean
"≥ 2/3 stake executed and accepted this block," not merely "signed these
roots." The permanence half depends on this as much as the privacy half —
invalid storage proofs must not be able to ride a finality quorum.

### F4. Encrypted threshold mempool (add)

Even with Dandelion++ (P3), the **mempool is public** to every producer and
relay node: tx timing, size, and graph topology leak before inclusion. Adopt
threshold-encrypted mempool admission (encrypt tx body to a rotating committee
key; decrypt only after the producer commits to inclusion order) so origin
metadata and inter-tx timing are hidden from the relay layer. This is the
missing link between on-chain RingCT and real-world network privacy — and it
protects permanence uploads from selective censorship at the mempool layer
(producers cannot drop storage-register txs they dislike if they cannot read
them early).

### F5. Fraud / validity proofs for light clients (add)

Until Tier 4 SNARKs (P11 path), ship **interactive fraud proofs**: any full
node can challenge a finalized block by posting a succinct witness that
`apply_block` would reject (bad CLSAG, bad SPoRA, bad coinbase, bad endowment).
Light clients treat "uncontested finality" as soft and "finality + no fraud
proof within N slots" as hard. Arweave never had this for money; Ethereum L2s
do — a permanence chain that outlives its full-node population **must**.
Natural home: new gossip type in `mfn-net` + slashing hook if the fraud proof
is valid (PM1 bonds).

### F6. Couple producer revenue to storage treasury health (change)

PROBLEMS #4: producers earn tail + fee tip even when the storage treasury
starves. Route a **basis-point coupling**: when treasury balance falls below
a slot-averaged runway threshold, temporarily shift fee share (or a tail slice)
from producer coinbase to treasury until runway recovers. Producers remain
incentivized to include blocks, but they cannot profit from a dying permanence
layer. One consensus parameter, enforced in
[`producer_coinbase_amount`](../mfn-consensus/src/emission.rs) — aligns the
two halves of the fused thesis without new cryptography.

### F7. Canonical input-count shapes — finish P5 (change)

The two-output floor is shipped; **input count still leaks**
(`tx.inputs.len()` is public). Enforce a small set of allowed `(inputs,
outputs)` shapes (e.g. always 2-in/2-out for standard transfers) via
consensus padding with zero-value inputs drawn from the wallet's own UTXO set
or explicit dummy inputs. Completes the "Monero default tx" fingerprinting
closure started in [`PRIVACY_HARDENING.md` §A1/§B1](./PRIVACY_HARDENING.md).

### F8. External checkpoint anchoring (add)

PM10 archives and PM16 broadcast checkpoints defend against **network** loss.
Add a low-rate, operator-run **anchor tx** that commits the latest finalized
checkpoint hash into another durable chain (Bitcoin OP_RETURN, Ethereum calldata,
or a second Permawrite shard). Permanence that survives only inside one ledger
is one geopolitical event away from rewrite. Cheap socially, high impact
psychologically — and it gives cold-start nodes a cross-check beyond genesis
trust.

### F9. Decoy-RNG entropy contract — finish P9 (change)

Conformance suite pins wire bytes; it does not yet pin **where** decoy RNG
bytes come from. Publish a normative contract: OS CSPRNG only, no user-supplied
seeds in production builds, log-once warning if `/dev/urandom` fails, WASM
uses `getRandomValues` exclusively. Test: two runs with the same wallet seed
must **not** produce identical decoy indices unless explicitly in test mode.
Partitions the anonymity set by wallet implementation otherwise.

### F10. Purge `f64` from consensus verification (change)

PROBLEMS #14: one `f64` survives on the consensus path. For a chain whose
docs promise multi-century horizons, floating point is an unacceptable
nondeterminism vector across architectures. Replace with fixed-point or
rational arithmetic in the cited path; add a CI lint that rejects new `f64` in
`mfn-consensus` / `mfn-storage` non-test code. Determinism is a permanence
property — light clients on WASM and mainframes must agree bit-for-bit.

## Part 6 — Privacy depth (additional)

### P17. Age-band coin selection — no era mixing (change)

> **Shipped.** `Wallet::select_inputs` groups spendable outputs into
> exponential age bands and spends within one band (fewest inputs; ties to
> the newest band). Spill across bands only when no single band covers the
> target. Stops input sets from mixing fresh and ancient UTXOs — a classic
> temporal fingerprint. See [`PRIVACY_HARDENING.md` §B2](./PRIVACY_HARDENING.md).

### P18. Tier 4: recursive SNARK block validity (add)

Batch-verify every CLSAG, Bulletproof, and balance equation in a block with
one constant-size proof (STARK or IPA recursion). Shrinks light-client
bandwidth, enables F5 fraud-proof compression, and is the performance
prerequisite for P11-scale membership proofs at mainnet UTXO counts. Document
as Tier 4 in the anonymity table in [`PRIVACY.md`](./PRIVACY.md).

### P19. Private stake metadata (add)

Validator pubkeys and stake weights are public today — fine for consensus, but
a wealth graph for anyone who bonds validators to real-world entities. Allow
optional **shielded stake pools**: users delegate into a ring-signed pool UTXO
that the pool operator bonds; delegators remain unlinkable from the validator
key on-chain. Raises the cost of targeting Permawrite's security budget
holders.

### P20. Uniform `extra` field and tx version pinning (change)

Any non-empty `tx.extra` is a public fingerprint. Consensus-reject non-canonical
`extra` (must be empty in production tier) except for a versioned, length-padded
extension block reserved for future upgrades. Extends the conformance suite in
[`canonical_conformance.rs`](../mfn-wallet/tests/canonical_conformance.rs).

### P21. Timing obfuscation for SPoRA proof submission (add)

Proof broadcast timing correlates operators to files even when payments are
private. Require operators to submit proofs through the same cover-traffic
regime as P16 (constant-rate proof slots, batching, padding). Reading and
*proving* should be as metadata-resistant as spending.

### P22. Default RPC and wallet transport hardening (change)

`PUBLIC_DEVNET_THREAT_MODEL.md` treats RPC exposure as operational risk;
for the privacy vision, **defaults matter**. Bind RPC to loopback, require
TLS or Unix socket for remote, reject unauthenticated `submit_tx` from non-local
peers, and document Tor hidden-service setup as the reference remote pattern.
Permanence operators who expose plaintext RPC leak their users' tx origins.

## Part 7 — Permanence depth (additional)

### PM18. Endowment top-up without re-anchoring (add)

Cold files whose treasury yield no longer covers SPoRA cost have no protocol
path to add funds. Add a `top_up_endowment` tx type: ring-signed transfer that
increments the same `commit_hash`'s endowment commitment (with opening/range
proof tied to F1). Lets communities fund archives, legal holds, and personal
memorials across centuries without duplicating `data_root` entries.

### PM19. Persistent proof obligation for cold data (change)

PROBLEMS #5: operators ignore unprofitable cold files. Scale minimum proof
frequency with file age so **every** anchored commitment continues to draw
challenges — not just hot uploads. Couple with PM12 repair bounties: if proofs
stop arriving for N windows, auto-escalate reward multiplier until PM1 slashing
or new operators appear.

### PM20. Geo/ASN-diverse replication scoring (change)

`replication: 3` does not prevent all three copies from sitting in one
availability zone. At proof reward time, weight payouts by **distinct ASN /
region** among provers for the same `data_root`; optionally require minimum
diversity before counting a window as satisfied. Permanence against datacenter
fire, regional war, and cloud-account seizure — not just single-disk failure.

### PM21. Chunk format v2: inline erasure + authenticated writes (add)

M5.49/M7.12 authenticated ChunkV1 gossip; extend to **ChunkV2** with
Reed–Solomon stripes (PM4) and a mandatory write signature bound to the
storage commitment. Operators store k-of-n fragments; a reader with any k
fragments reconstructs. Combines PM4 + PERMANENCE_HARDENING replication
accounting into one wire upgrade.

### PM22. Treasury runway oracle and public alerts (add)

Fee drought (PROBLEMS #2) is silent until proofs fail. Expose a chain-derived
**runway metric** (treasury balance / trailing payout rate) in RPC and light
client UI; trigger wallet warnings when runway falls below configurable
thresholds. Economic transparency does not weaken privacy — amounts are
aggregate, not per-user. Gives holders and archivists time to top-up (PM18) or
advocate for F6 coupling before permanence degrades.

### PM23. Immutable operator storage manifest (add)

Each bonded operator (PM1) publishes a signed **manifest** of
`(data_root → chunk indices held, disk epoch, erasure params)` updated only
on add/prune; peers gossip manifests and light clients spot-check against
SPoRA winners. Creates accountability below the proof layer — an operator
cannot silently drop a file they still claim in manifest without being
detectable before slashing.

### PM24. Time-locked protocol upgrade veto (add)

PM13 ossifies parameters; this ossifies **process**. Multi-year timelock +
supermajority for any change that lowers ring size, tail emission, storage
prune carve-out, or endowment binding. Prevents a captured validator set from
strip-mining permanence or privacy in an emergency fork. Complements
[`constitution.rs`](../mfn-consensus/src/constitution.rs) with governance
latency as a feature.

### PM25. Continuous differential fuzzing of `apply_block` (add)

PM14's second client is the gold standard; until it ships, run 24/7
differential fuzz (Rust reference vs. minimal spec interpreter, or
structure-aware proptest mutations) in CI nightly, seeded from
[`docs/interop`](./interop) and M5 mixed CLSAG+storage vectors. Permanence
means no silent inflation bug can hide for a quarter — fuzzing is the
cheapest infinite-horizon insurance policy short of PM15 proofs.

## Part 8 — Adversarial resilience (state-level and above)

These items assume an adversary with ISP visibility, jurisdictional capture,
cloud-account seizure, or long-run archival — not just a chain analyst.

### P23. Censorship-resistant block inclusion for paid storage anchors (change)

A rational producer can silently drop storage-register txs that pay full fee
while still collecting tail emission on empty blocks — a **permanence
censorship** vector distinct from payment censorship. Require producers to
include all fee-valid storage-register txs from the mempool up to a per-block
byte budget (or forfeit a slice of producer coinbase / face slashing evidence
if a excluded tx is later included by the next producer). Touches
[`mfn-runtime/src/producer.rs`](../mfn-runtime/src/producer.rs) and block
template policy. Pairs with F4 (encrypted mempool): inclusion rules must not
depend on reading tx plaintext early.

### P24. Metadata minimization on storage commitments (change)

Beyond P15 (size buckets), **`replication` and exact `chunk_size` are public
fingerprints** of operator strategy and file class. Move to canonical tiers:
allowed `(chunk_size, replication)` pairs only (e.g. 256 KiB × 3, 1 MiB × 5),
consensus-enforced at anchor time. Uploaders still pay `required_endowment` on
true `size_bytes`; padding fills the bucket. Same privacy argument as uniform
ring size — fewer public degrees of freedom.

### P25. In-wallet churn / merge transactions (add)

Output count and UTXO age distribution drift over years of use. Ship a
reference **churn** workflow: periodic self-to-self transfers at canonical
shape (F7), scheduled with cover traffic (P16), using age-band selection
(P17). Keeps the UTXO set healthy for Tier 3 membership proofs and reduces
long-run graph analysis surface without a third-party coinjoin coordinator.

### P26. Constant-time and side-channel-hardened crypto builds (add)

RingCT privacy is computational; microarchitectural leakage is not. Publish
**hardened build profiles** (`RUSTFLAGS` for constant-time where the curve
libs support it, disable hyper-threading guidance for signing processes,
WASM run in isolated worker). Add a CI job that runs signing/verify under
dudect-style timing tests on CLSAG verify. Nation-state adversaries target
implementations, not just protocol math.

### P27. Subaddress rotation and payment-request hygiene (change)

Indexed subaddresses reduce linkability for merchants but rot poorly by
default. Wallet policy: rotate receiving subaddress index on a schedule or
after N deposits; never reuse `(R, P)` pairs across unrelated counterparties;
payment URIs include subaddress index + optional amount bucket (not exact
amount in URI when avoidable). Document in wallet README and enforce in CLI
`wallet receive` output. Reduces long-horizon address-reuse deanonymization.

### P28. ZK-wrapped SPoRA submission (add)

Today the proof pool carries **chunk bytes in the clear** to producers
(PROBLEMS historical note on producer advantage). Long-term: prove chunk
Merkle inclusion + SPoRA response in a ZK wrapper so mempool observers and
non-winning producers learn nothing about which file is being proved until
inclusion. High effort; unblocks private operator topology (home operators
cannot be filtered by chunk content at relay time).

## Part 9 — Century-scale operations and supply chain

Permanence and privacy fail in practice if releases, genesis ceremonies, and
operator tooling are brittle — regardless of on-chain math.

### PM26. Genesis BLS proof-of-possession ceremony (change)

PROBLEMS #13: genesis validators skip PoP; rogue-key attacks poison the
initial committee if genesis is assembled from third-party submissions. Require
PoP for **every** genesis validator in `genesis_config_from_json_bytes` and in
public-testnet ceremony tooling (`scripts/public-devnet-v1/`). Same bar as
`BondOp::Register` in [`bond_wire.rs`](../mfn-consensus/src/bond_wire.rs).
A forever chain starts once; genesis is the highest-leverage trust event.

### PM27. Strip legacy validator-less mode from production builds (remove)

PROBLEMS #16: empty validator set silently sinks producer fee share and
emission. Feature-gate `legacy mode` to `cfg(test)` / dev-only binaries so
no production `mfnd` can accidentally run a value-destroying configuration.
Shrinks the attack surface and the "undocumented economics" trap.

### PM28. Incentivized archive-full-node role (add)

Light clients and PM10 exports depend on someone retaining **full blocks +
all chunk bytes**. Storage operators optimize for SPoRA-hot data; archive nodes
optimize for completeness. Fund a tail-emission slice or treasury micro-reward
for nodes that serve historical chunk ranges provably (hash challenge over
random historical `(data_root, index)` pairs). Without this, permanence
 degrades into "recent data only" as operator economics dominate.

### PM29. Anti-permanence-spam: minimum utility density (change)

Anyone can anchor garbage forever if they pay once. Add a **consensus fee
floor per byte** escalator tied to chain storage occupancy (similar to EIP-1559
for permanent bytes): as total anchored bytes grow, `required_endowment` scales
superlinearly for new uploads. Prevents state/hosting capture by junk anchors
without touching existing commitments. Complements PM8 pruning carve-out —
even pruned money-state chains still pay for data state.

### PM30. Operator unbond delay with proof tail (change)

When PM1 bonds exist, unbonding must not be instant — operators could bond,
accept uploads, earn, and exit before the first SPoRA window catches defectors.
Require `unbond_delay_slots` during which the operator must **continue proving**
all manifests (PM23) or forfeit bond. Mirrors validator unbond semantics in M1.

### PM31. P2P seeding of `archive-export` bundles (add)

PM10 produces portable archives; PM16 broadcasts checkpoints. Add optional
**BitTorrent/IPFS/WebTorrent** seeding of export bundles keyed by genesis hash +
tip checkpoint, initiated from `mfnd archive-export`. Permanence beyond any
single organization's bandwidth — the same way Linux distros survive.

### F11. Reproducible builds and signed release transparency (add)

Supply-chain compromise breaks every privacy guarantee (malicious wallet steals
seeds). Ship **reproducible** release artifacts for `mfn-cli`, `mfnd`, and
`mfn-wasm`, publish signed hashes in-repo and on a transparency log, and
document verifier steps in [`CI.md`](./CI.md). Users who verify binaries match
audited source are part of the privacy story — not optional ops hygiene.

### F12. Subjective checkpoint web for light clients (add)

Cryptographic finality plus PM16 broadcast is necessary; **social redundancy**
is also necessary. Maintain a signed, append-only **checkpoint log** (maintainer
keys + community witnesses) that light clients can cross-check against P2P tip.
If a fork violates PM13 constitution, clients reject it even with valid-looking
BLS — governance capture resistance for wallets that will outlive any one
foundation.

### F13. Threshold custody for institutional archives (add)

Museums, newsrooms, and dissident archives need **N-of-M spend** without
on-chain multisig that labels the institution. Research threshold RingCT /
FROST-style threshold CLSAG for `top_up_endowment` (PM18) and storage anchor
 txs so no single compromised key can redirect or starve a corpus. Privacy +
 organizational permanence — neither pure multisig (transparent) nor pure
 single-key (fragile) suffices.

### F14. View-only delegation with minimal leakage (add)

View keys enable audit but handing them to a third party exposes full incoming
history. Add **scoped view capabilities**: subaddress-scoped or time-bounded
view tags derived keys that reveal only a subset of outputs. Lets accountants,
lawyers, and heirs participate without permanent full-history exposure —
critical for multi-generational permanence use cases.

### F15. Document and test the MFBN-1 VRF variant (change)

PROBLEMS #15: Permawrite VRF is not RFC 9381-interop. Publish the exact
MFBN-1 spec in [`docs/interop`](./interop), ship test vectors, and gate any
claim of "standard VRF" until either documented as MFBN-1 or migrated to strict
Elligator2. Interop clarity is a permanence property — future tooling must not
guess hash-to-curve behavior wrong for decades.

## Part 10 — Research horizon (no shortcut around physics)

Items here are **research-gated** but belong on the menu so the vision does not
quietly cap at "Monero + Arweave."

### P29. Seraphis / Triptych alignment path (add)

Track Monero's successor signature research (Seraphis, FCMP++) and maintain a
**compatibility migration** branch in `mfn-crypto` so Permawrite does not fork
into a dead-end curve tree while the field converges. P11 is the target; P29
is the process discipline.

### P30. Deniable wallets (optional second passphrase) (add)

Plausible deniability for wallet files at rest: decoy wallet decrypts with
password A, real wallet with password B. Does not change chain privacy; protects
users when coerced at the device layer. Reference implementation in
`mfn-wallet` keystore format only — optional, off by default.

### PM32. Endowment outlives operator set collapse (simulation + policy) (add)

Model the **absorbing state**: what happens if operator count → 0 but data
remains anchored? PM12 repair bounties + PM19 cold proofs + PM28 archive nodes
should be simulated together in emission tests (`emission_simulation.rs`) with
published break-even operator counts. Permanence claims need numeric tail-risk
bounds, not narrative.

### PM33. Physical media export profile (add)

Extend PM10 export with a **gold-media profile**: fixed block size, maximal
forward-error correction, human-readable manifest header, QR-encoded genesis
hash — suitable for M-DISC / tape deposit. Permanence that survives network +
software extinction requires thinking about atoms, not just blocks.

## Part 11 — Network layer and participant hardening

On-chain RingCT is necessary but not sufficient. These items close gaps between
**protocol privacy/permanence** and **what actually runs on someone's laptop**.

### P31. Anti-eclipse peer diversity (change)

A node that sees only adversary-controlled peers learns which txs you submit
and which chunks you serve. Enforce minimum **outbound peer diversity** in
`mfn-net`: distinct /16 (or ASN when available), minimum honest-relay count,
anchor peer list shipped with checkpoints (F12), and automatic redial when
diversity drops. Eclipse attacks are the fastest way to deanonymize Dandelion++
(P3) and to partition operators from producers (PM3 lottery path).

### P32. Role-separated node topology (add)

`DECENTRALIZATION.md` splits validators, operators, wallets, and observers —
but nothing stops one VPS from running all roles on one IP, correlating spend
and serve behavior. Document and ship **reference topology**: wallet keys never
on operator machines; operator manifests (PM23) never on wallet machines;
validators on dedicated hosts. Add `mfnd` startup lint: warn when operator +
validator + RPC wallet share the same advertised listen address.

### P33. Hardware wallet and secure-enclave signing (add)

Software-only spend keys on internet-connected laptops fail the "state actor"
bar. Integrate a signing backend abstraction in `mfn-wallet` (USB HW wallet,
TPM, Secure Enclave via platform APIs, airgap QR) so CLSAG construction runs
on-host but **secret scalars never leave the secure device**. Permanence users
hold large endowments; privacy users are high-value targets — both need HSM-grade
signing paths, not only exchange-grade hot wallets.

### P34. Uniform fee buckets (change)

Fees are public (P6 is the long-term fix). Until hidden fees ship, enforce
**wallet-side fee quantization** to a small set of bucket sizes (e.g. powers
of two in base units) so fee values cannot fingerprint wallet brand or tx
type. Extend [`canonical_conformance.rs`](../mfn-wallet/tests/canonical_conformance.rs)
with fee-bucket pins; optional future step: consensus minimum fee granularity.

### P35. Wallet state encryption and memory hygiene (change)

Keystore encryption is table stakes; add **mandatory** Argon2id parameters,
secure zeroization of scalars after sign (`zeroize` crate audit across
`mfn-wallet` / `mfn-crypto` signing paths), and WASM worker isolation so
browser wallets do not share a heap with untrusted page JS. A chain that
promises forever keys must not leak them through a buffer reuse bug in a
five-year-old tab.

### P36. Private bootstrap and checkpoint fetch (add)

Initial sync currently trusts whoever serves genesis + blocks first. Support
bootstrap exclusively over Tor (P4), checkpoint-first sync (F12), and
**digest-pinned** genesis bundles (`expected_genesis_hash` required flag on
`mfnd serve`). Prevents ISP-level feeding of a forked or stripped chain to
new participants — an attack that defeats both privacy (wrong UTXO set) and
permanence (missing storage state).

### PM34. Cryptographic proof-of-replication (add)

SECURITY_CONSIDERATIONS §8.4: SPoRA proves *an* copy exists, not `replication`
distinct copies on independent failure domains. Add an optional **multi-operator
proof bundle**: per window, require proofs from N distinct operator pubkeys
(PM2) whose manifest entries (PM23) cover the same `(data_root, chunk_index)`
before the window counts as satisfied. Without this, replication is pricing
theater — three paid copies can be one disk with three keys.

### PM35. Unbiased SPoRA index sampling (change)

[`chunk_index_for_challenge`](../mfn-storage/src/spora.rs) uses raw `mod
num_chunks`; bias is negligible today but permanence docs outlive IEEE-754
arguments. Replace with rejection sampling (or wide reduction) so challenge
index is exactly uniform for all `num_chunks`. One function, zero protocol
controversy, removes a footgun for future auditors.

### PM36. Operator Sybil resistance via bonded manifests (change)

When PM1 bonds ship, tie **one manifest identity per bond** and rate-limit
new manifest pubkeys per bond per epoch. Prevents one operator from simulating
geo-diversity (PM20) with Sybil keys while storing a single copy. Slashing
trigger: two manifests with same bond hash prove conflicting chunk presence
for the same slot.

### PM37. Raspberry-Pi-class operator reference profile (add)

Permanence decentralization requires **millions of small operators**, not dozens
of datacenters. Publish a tested minimum profile (ARM SBC + USB disk, RAM,
bandwidth) and a `mfn-storage-operator --profile=minimal` preset in
[`STORAGE_ACCESSIBILITY.md`](./STORAGE_ACCESSIBILITY.md). Run it in CI nightly
on a low-resource job. If the reference stack cannot run on Pi-class hardware,
the permanence vision is false advertising.

### PM38. Trust-minimized snapshot sync (add)

Full sync from genesis is a permanence bootstrap barrier. Extend
[`chain_checkpoint`](../mfn-consensus/src/chain_checkpoint) with **witnessed
snapshots**: state root + storage root + block height signed by finality quorum,
verifiable by light clients without replaying every block. Pairs with PM8
(prune behind checkpoint) and F12 (social checkpoint web) — fast join without
trusting a single RPC's `get_block`.

### PM39. Periodic global SPoRA audit lottery (add)

Hot files get proved often; **cold files can silently rot** (PROBLEMS #5).
Once per epoch, draw a small set of `(commit_hash, chunk_index)` pairs from
the entire anchored set (VRF from epoch seed) and elevate reward + slash
pressure for those proofs (PM19 + PM12). Makes "every byte forever" a
scheduled obligation, not a hope that operators stay honest on unpopular data.

### F16. Mutual authentication on validator P2P (add)

Validator mesh traffic carries block proposals and votes — metadata useful for
traffic analysis and injection attacks. Require **mutual TLS or Noise IK** on
validator peering (distinct from public observer P2P), with keys rotated via
on-chain validator registry. Consensus privacy is not RingCT, but unauthenticated
P2P is how eclipse and partition attacks start.

### F17. Privacy-preserving node telemetry (change)

Operators and observers need metrics; metrics must not become surveillance.
Define an RPC **aggregate-only** stats surface (treasury runway PM22, proof
rates, peer counts) and forbid per-peer / per-tx logging in default configs.
Document redaction rules in [`PUBLIC_DEVNET_THREAT_MODEL.md`](./PUBLIC_DEVNET_THREAT_MODEL.md).
A permanence network where every observer logs IP ↔ tx hash is a honeypot.

### F18. Privacy/permanence impact gate in CI (add)

Any PR touching `mfn-wallet`, `mfn-node` mempool, or `apply_block` storage
path must include a one-paragraph **F5 impact note** in the PR template:
does it weaken ring policy, endowment binding, or SPoRA verification? Automate
a lightweight check in `scripts/ci-check` that fails if those paths change
without updating `PRIVACY_HARDENING.md` or `PERMANENCE_HARDENING.md`. Process
as code — the doctrine in [`AGENTS.md`](../AGENTS.md) already demands it.

## Part 12 — Measurement, accountability, and honest limits

Absolute privacy and permanence require **knowing what is still open** and
publishing numbers — not marketing adjectives.

### P37. Public anonymity-set telemetry (add)

Publish weekly **aggregate** chain stats: UTXO count, median output age,
fraction of txs at canonical shape (once F7 ships), ring-size compliance rate
(should be 100% under production params). No per-address data. Lets the
community detect wallet-ecosystem drift (implementation partition) before
attackers exploit it. Monero ships similar reports; a privacy chain that refuses
to measure its anonymity set is flying blind.

### P38. Wallet implementation diversity target (add)

Client monoculture (PM14) applies to wallets too. Track and publish counts of
**distinct wallet software versions** seen via tx conformance fingerprints
(P20, P34) — again aggregate only. Set a project goal: no single frontend
> 80% of txs for 90 days. Encourages third-party wallets and catches accidental
canonical drift in the reference implementation.

### PM40. Permanence SLA dashboard (add)

Expose operator-facing and public dashboards: % of `data_root`s meeting
replication target, cold-file proof latency (PM39), treasury runway (PM22),
mean time-to-repair (PM12). Numbers grounded in chain state + manifest gossip
(PM23), not operator self-report. "More permanent than anything on earth" must
be falsifiable — this is how you falsify it.

### PM41. Emission backstop circuit breaker (change)

When treasury is dry, `storage_proof_reward` mints fresh tokens — correct
backstop, but unbounded mint is a permanence **and** monetary catastrophe.
Cap backstop mint per epoch as a function of tail emission (e.g. ≤ 1% of
annual tail per epoch) and log when the cap binds. Simulated in
[`emission_simulation.rs`](../mfn-consensus/tests/emission_simulation.rs)
alongside PM32 tail-risk scenarios.

### PM42. Replication accounting in consensus (add)

PERMANENCE_HARDENING documents authenticated chunk gossip; extend with
**on-chain replication counters**: per `commit_hash`, count distinct operator
pubkeys that proved in the last K windows (sourced from accepted proofs in
`apply_block`). RPC exposes `replication_health(commit_hash)` for uploaders
and PM40 dashboards. Closes the loop between PM2 policy and observable reality
without waiting for PM34's cryptographic bundle.

### F19. Decentralization index (add)

Single metric combining stake Gini (validators), operator ASN diversity (PM20),
archive-node count (PM28), and geographic spread of manifest endpoints —
published monthly from aggregate P2P data. [`DECENTRALIZATION.md`](./DECENTRALIZATION.md)
describes roles; F19 measures whether the network matches the doc. Targets
(e.g. no validator > 15% stake) become explicit, not aspirational.

### F20. Adversarial testnet program (add)

Permanent **red-team net** (distinct from public devnet) where paid researchers
attempt eclipse, producer censorship (P23), storage withholding, and fork
feed attacks against RC builds. Findings feed F5 and `PROBLEMS.md` with bounty
payouts from PM17 protocol endowment. Privacy/permanence claims without
continuous adversarial rehearsal are untested claims.

### F21. User-facing threat-model wizard (add)

Ship a 5-minute interactive flow in docs/WASM: "What are you storing? Who is
your adversary?" → outputs required mitigations (F2 encryption, P13 private
reads, F13 threshold custody, subaddress rotation P27). Absolute privacy fails
when users upload plaintext whistleblower docs because nobody told them to
encrypt — UX is a privacy layer.

## Part 13 — Sovereignty, composability, and human-rights operations

These items address **capture, coercion, and jurisdiction** — adversaries who
cannot break CLSAG but can pressure operators, seize domains, or outlaw clients.

### P39. Commit–reveal storage registration (change)

Upload txs currently expose `data_root` and metadata at mempool time — before
inclusion, producers and relays can censor specific content. Two-phase anchor:
(1) ring-signed commit to `H(data_root ‖ salt)` only; (2) reveal in a later
block after commit is final. Reveal without commit is rejected; commit without
timely reveal expires and frees mempool space. Censorship requires blocking
*all* commits, not selected hashes. Touches storage-register tx type in
`mfn-consensus` and wallet upload flow.

### P40. Payment–storage unlinkability policy (add)

A user who pays endowment from UTXO **A** and later spends change from the same
tx graph links permanence to identity. Ship wallet **policy profiles**:
`storage-isolated` (fresh subaddress + dedicated input band for uploads only,
never mixed with daily spend UTXOs), documented in wallet README and F21 wizard.
Privacy and permanence fused on-chain; unlinkability still fails if the wallet
reuses the same pot.

### P41. Onion-service prove+serve endpoints (add)

P13 private reads need transport, not only crypto. Require reference operators to
publish `.onion` (or equivalent) serve URLs in manifests (PM23), and teach
`mfn-cli` / WASM fetch to prefer hidden-service paths over clearnet IP. Reading
a permanence archive over the operator's logged IP defeats the point for
dissidents in hostile networks.

### P42. Block-producer transaction ordering privacy (change)

Even with encrypted mempool (F4), the producer learns inclusion order. Use
**order-preserving encryption** or a VRF shuffle of decrypted txs immediately
before block assembly so the producer cannot sell MEV on storage-register
position or target adjacent placement attacks. High effort; pairs with P23
mandatory inclusion.

### PM43. Slash-funded legal defense and operator protection pool (add)

PM1 slashing sends bonds to treasury — good for economics, insufficient for
operators raided for hosting controversial permanence. Allocate a **fixed bps**
of slash inflows to a defense pool (multisig spend policy) funding legal aid,
server replacement, and PM16 checkpoint mirrors for operators facing seizure.
Permanence requires operators willing to hold unpopular data; without backstop
support, economics favor only mainstream bytes.

### PM44. Scheduled reveal for encrypted archives (add)

F2 encrypt-by-default creates **key-death** risk: ciphertext is permanent but
keys die with the author. Optional `reveal_slot` on anchor: ciphertext on-chain
from day one, plaintext key release enforced by consensus at future slot
(dead-man switch, embargo lift, estate plan). Combines permanence (bytes exist)
with privacy (bytes meaningless until reveal). Wallet and `apply_block` must
reject early key publication.

### PM45. Sovereign state export for chain capture (add)

If governance capture breaks PM13 despite client ossification, users need an
**exit artifact**: one command produces a minimal verifiable bundle (final
checkpoint + all storage commitments + chunk Merkle paths + UTXO accumulator
snapshot) provable without trusting the live chain tip. Extends PM10 archive
export with a "capture-response" profile documented for human-rights orgs.
Permanence means the bytes survive **even if the chain social consensus fails**.

### F22. Jurisdiction-aware operator guidance (add)

Map legal regimes (EU, US, etc.) to **operator obligations** without pretending
the chain can enforce law: what encryption + role separation (P32) achieves,
what it cannot, when to run onion-only (P41), when PM43 pool applies. Lives in
`scripts/public-devnet-v1/OPERATORS.md` and F21 wizard — not legal advice, but
honest limits beat false confidence for archivists.

### F23. Third-party read-only verification SDK (add)

Journalists and courts should verify permanence **without running mfnd**. Ship a
minimal `@permawrite/verify` (WASM/JS) that checks: genesis constitution (PM13),
checkpoint signatures, `data_root` Merkle path, SPoRA proof for a sample chunk.
No wallet, no P2P — just "does this hash match what the chain committed?"
Composability is permanence: data outlives any one client binary.

### F24. Post-quantum Merkle hash agility (add)

PQ_MIGRATION covers signatures and stealth; **Merkle trees still assume SHA-256**
for centuries. Reserve `hash_alg` byte in `StorageCommitment` and block header
roots; document upgrade to SHA-3 / SLH-DSA-era hash without re-anchoring data
(dual-root transition blocks). A permanence chain must plan hash agility, not
only signature agility.

## Part 14 — Scale without dilution

Growth that weakens ring policy, endowment binding, or SPoRA verification is
anti-goals. These items add capacity **only** inside the doctrine.

### P43. Batch ring verification at block level (add)

At high tx volume, per-tx CLSAG verify dominates block cost. Implement
**batch CLSAG verification** (Straus / multi-scalar strategies) in
`mfn-crypto` and call from `verify_transaction` block ingress. Does not change
privacy; reduces the excuse to shrink rings or skip proofs under load.

### P44. Aggregated storage uploads (change)

Many small files → many on-chain anchors → state bloat. Allow **one tx** to
register N storage commitments with shared ring signature overhead and one
fee, while each commitment still carries independent endowment (F1-bound).
Wallets batch museum digitization dumps without N separate CLSAGs.

### P45. Scan sharding for light wallets (add)

View-key scanning is O(outputs × blocks). With view tags (P7), add **height-range
subscription**: wallet tracks only subaddress subset + block range, fetching
ciphertext blobs via oblivious sync (P14). Keeps mobile verification cheap at
billions of outputs without resorting to trusted full-node scan services.

### PM46. Erasure-coded cold tier with hot SPoRA cache (add)

PM4 erasure coding plus operator economics: **hot tier** (full chunk, frequent
SPoRA) + **cold tier** (erasure fragments on tape/PM33 media, slower audit
lottery PM39). Endowment pricing reflects tier; PM12 repair promotes cold→hot
when hot copies vanish. Scales bytes stored per operator disk without pretending
one SSD holds the world.

### PM47. Stateless client verification path (add)

PM38 snapshots help nodes join; **stateless verification** (witness for UTXO
+ storage reads per block) helps observers verify one block without full
history. Research track aligned with P18 SNARKs — document in ROADMAP as
scale enabler, not a privacy downgrade (witnesses must not reveal spend keys).

### PM48. Capacity-aware endowment curve (change)

PM29 anti-spam escalator is reactive. Proactive: publish **`required_endowment
= f(total_anchored_bytes, treasury_runway)`** with on-chain parameters voted
under PM24 timelock, simulated in emission tests (PM32). Prevents sudden
capacity cliffs that force rushed forks or fee spikes that kill privacy usage.

### F25. Throughput floor SLO in release gates (add)

Add to [`TESTNET_CHECKLIST.md`](./TESTNET_CHECKLIST.md): RC must publish
**verified TPS** for privacy txs and storage anchors separately on reference
hardware, with rings=16 and two outputs. Scale claims must be measured before
mainnet — otherwise operators plan for throughput that collapses under real
CLSAG load, and users flee to transparent chains.

### F26. Green permanence metrics (add)

Permanent bytes imply permanent disk spin — publish **energy per anchored
gigabyte-year** for reference operator profiles (PM37 Pi, datacenter tier).
Not greenwashing: erasure coding (PM4/PM46) and lottery proofs (PM3) should
show lower joules/byte than naive triple replication. Societal legitimacy
matters for century-scale infrastructure.

### F27. Accessibility parity for privacy UX (add)

[`UX_ACCESSIBILITY.md`](./UX_ACCESSIBILITY.md) applies to wallets too: screen
reader support for CLI confirmations, high-contrast WASM wallet, keyboard-only
signing flows. Privacy tools used only by terminal elites fail the human-rights
brief — dissidents include blind and low-vision users.
